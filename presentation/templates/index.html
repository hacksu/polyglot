01 — Definition

What is a Polyglot File?

A polyglot file is a single file that is simultaneously valid in two or more languages or formats — each parser reads the same bytes and produces correct, meaningful output.

Core insight

  • Not a bug — exploits formal grammar rules
  • Every language has regions it ignores
  • Hide language B inside language A's ignored regions
  • 100% valid in both languages simultaneously
  • The bytes never change — only the parser does

What we'll cover today

  • HTMLCSSJavaScript — already in every webpage
  • JPEGZIP — binary format tricks
  • CPython — the preprocessor trick
  • BashPython — the exec trick
  • PythonJavaScript — polycompiler library
02 — Parser Mechanics

How Do Parsers See the World?

Every language has regions it ignores. Polyglots hide language B inside language A's ignored regions — and vice versa.

Comments

  • #  Python, Bash, Ruby, Perl
  • //  C, C++, Java, JavaScript
  • /* */ C, CSS, Java
  • <!-- --> HTML, XML
  • --  SQL, Lua, Haskell

Skip Regions

  • #if 0 … #endif in C
  • """...""" string literals (Python)
  • /* ... */ block comments
  • heredocs in Bash / Ruby
  • Shebang lines  #!/usr/bin/env

File Format Rules

  • JPEG reads from the start
  • ZIP reads from the end
  • PDF has flexible internal structure
  • Magic bytes checked in first N bytes
  • Trailing data is often ignored
The rule: If parser A ignores region R, and region R is valid in parser B, you can hide language-B code inside R — and vice versa.
03 — HTML + CSS + JavaScript

The Web Is Already Polyglot

Every .html file is a polyglot — three parsers, one file, by design.

<!DOCTYPE html>
<html>

  <!-- HTML PARSER reads the structure -->

  <style>
    /* CSS PARSER handles styles */
    body {
      background: #0d0d0d;
      color: var(--green);
      font-family: monospace;
    }
  </style>

  <script>
    // JAVASCRIPT ENGINE runs logic
    document.addEventListener('DOMContentLoaded', () => {
      console.log("three parsers, one file");
    });
  </script>

  <p>Hello, polyglot world!</p>

</html>

HTML Parser

Reads tags, builds the DOM tree. Treats <style> and <script> contents as opaque blobs and hands them off.

CSS Parser

Only processes what's inside <style> tags. The surrounding HTML is completely invisible to it.

JS Engine

Only runs what's inside <script> tags. Has no awareness of HTML or CSS syntax around it.

04 — Binary Polyglots

Binary Polyglots — Images

File format parsers have their own ignored regions. A single file can be a valid JPEG and a valid ZIP archive simultaneously.

┌──────────────────────────────────────┐ │ polyglot.jpg (on disk) │ ├──────────────────────────────────────┤ │ FF D8 FF E0 ... ← JPEG magic bytes │ │ [JPEG image data ................] │ │ [JPEG image data ................] │ │ FF D9 ← JPEG end marker │ ├──────────────────────────────────────┤ │ PK\x03\x04 ... ← ZIP local header │ │ [ZIP file entries ................] │ │ PK\x05\x06 ... ← end-of-central- │ │ directory │ └──────────────────────────────────────┘
Image viewer: reads from the start → sees valid JPEG ✓
ZIP tool:    reads from the end  → sees valid ZIP ✓
Neither parser notices the other's data.

Why it works

  • JPEG spec: parse forward from magic bytes
  • ZIP spec: find end-of-central-dir from the back
  • JPEG ignores all data after FF D9
  • ZIP ignores bytes before its local headers

Real-world examples

  • CTF forensics — classic puzzle type
  • GIFAR — GIF+JAR (Java applet in image)
  • PDF+ZIP — common in competitions
  • Malware — evade file-type content filters
Try it: cat image.jpg archive.zip > polyglot.jpg
Open in an image viewer, then run unzip polyglot.jpg
05 — The C + Python Trick

The C + Python Mechanism

We exploit two features: C's #if 0 … #endif preprocessor directive and Python's """…""" triple-quoted strings.

C PREPROCESSOR
#if 0    ← skip until matching #endif
x = """
#endif   ← resume normal compilation
#include <stdio.h>
int main(void) {
  printf("Hello from C!\n");
}
#if 0    ← skip again
"""
print("Hello from Python!")
#endif
PYTHON INTERPRETER
#if 0    ← Python comment, ignored
x = """  ← triple-quoted string begins
  #endif
  #include <stdio.h>
  int main(void) {
    printf("Hello from C!\n");
  }
  #if 0
"""     ← string ends, x assigned
print("Hello from Python!")
#endif   ← Python comment, ignored
C never sees the print() call.  Python never executes the printf(). Both are 100% correct in their own language.

07 — Bash + Python

Bash + Python: The Exec Trick

Uses the difference between quoted words in bash and adjacent string literals in Python. 

{{ bash_python }}

Bash sees line 6:

"exec" → the exec shell builtin
"python3" "$0" "$@" → its arguments

Shell is replaced by Python running this file. Bash never reaches line 8+.

Python sees line 6:

"exec" "python3" "$0" "$@"
→ four adjacent string literals
→ concatenated: "execpython3$0$@"
→ expression statement, result discarded

No exec happens. Python runs the import + print below.

Key: Python's $0 and $@ are literal strings — no shell expansion. Bash's quoting preserves the builtin name.
08 — polycompiler

polycompiler — Python + JavaScript

A library by evanzhoudev that merges separate Python and JS files into one file runnable by both CPython and Node.js.  github.com/evanzhoudev/polycompiler

Install + use

npm install -g polycompiler

# Merge Python + JS into one file
polycompiler script.py script.js -o both.py

# Same file, two runtimes:
python both.py
node   both.py

Why Python+JS is harder

Python and JavaScript share similar surface syntax — you can't just hide one inside the other's comment blocks. polycompiler uses semantic divergence: expressions that evaluate differently in each runtime to branch execution.

script.py script.js │ │ └──── polycompiler ──┘ │ both.py ┌────┴────┐ │ │ python3 node runs it runs it

How it branches

  • Detects runtime via expressions that differ
  • (1).__class__.__name__ → 'int' in Python, error in JS
  • typeof undefined → valid in JS, NameError in Python
  • Uses these divergences to route to the right code path
