Slide 1

Title - Polyglot Programming

  • Welcome everyone. Ask the room: "How many of you have ever written code that compiles in two different languages at the same time?" - usually zero hands.
  • Today we'll build to a place where that answer is yes, and where you can explain exactly why it works.
  • Point at the two terminal commands on screen - python demo.c and gcc demo.c -o demo && ./demo - tell them both actually run right now from the same unmodified file.
  • The goal isn't magic tricks. It's understanding parsers deeply enough to exploit them deliberately.
Slide 2

What is a Polyglot File?

  • Stress: this is not a hack or a bug. It follows the formal grammar rules of both languages exactly.
  • Key mental model: every parser has regions it ignores. A polyglot is just code that hides language B inside language A's ignored regions.
  • The bytes on disk never change - only the parser changes.
  • Walk through the "what we'll cover" list so students know where the lesson is going.
Slide 3

How Do Parsers Work?

  • Ask the room: "What characters start a comment in Python? In C? In HTML?" Get them thinking about how different every language is.
  • Point out the three categories: syntax-level comments, skip directives (like #if 0), and file-format-level rules (JPEG/ZIP).
  • The golden rule at the bottom is worth reading aloud and pausing on: "If parser A ignores region R, and R is valid in B, you can hide B inside R."
  • This one rule explains every example that follows.
Slide 4

The Web Is Already Polyglot

  • This is the "aha" slide for web students - they've written polyglot code every day without knowing it.
  • Walk through the code left-to-right: HTML parser hands off <style> to the CSS parser and <script> to the JS engine. Three parsers, one file, by design.
  • The CSS parser literally cannot parse HTML syntax. It only sees what's inside <style> tags. Everything outside is invisible to it.
  • Point out: browser makers chose this design intentionally - they needed three languages in one file for the web to work.
Slide 5

More Examples

  • Quick tour of all four patterns side-by-side so students can see the variety.
  • HTML+Python uses a double trick: """...""" hides HTML from Python, and <!-- --> comments hide Python from the browser.
  • Shell+Python is the examples/bash_python.sh they can run right now.
  • Python+JS is what polycompiler outputs - coming up in slides 10 and 11.
  • C+Python is the main demo - coming up in slides 7 and 8.
Slide 6

Binary Polyglots - Images

  • This is the most surprising one for most students - no source code involved at all.
  • Key insight: JPEG reads from the start, ZIP reads from the end. They never step on each other.
  • The simple concatenation cat image.jpg archive.zip > both.jpg literally works - try it live if you want.
  • GIFAR was a real browser vulnerability (2008) - a GIF+JAR file could run Java code while appearing to be an image, bypassing same-origin policy. It was patched but the concept lives on in CTF challenges.
  • Malware uses this today to evade antivirus scanners that check file magic bytes.
Live demo option: grab any JPEG, zip a text file, concatenate them, verify both open image and unzip work on the result.
Slide 7

The C + Python Mechanism

  • Walk both columns simultaneously, line by line. Point to the left then right for each line.
  • C side: #if 0 is a preprocessor directive. The preprocessor runs before the compiler even sees the code. It strips out the entire block. The compiler never knows the Python code existed.
  • Python side: #if 0 is just a hash comment - the # character makes the rest of the line invisible to Python. Then x = """ opens a triple-quoted string and everything until the closing """ is just string data assigned to x.
  • Neither language produces an error. Both produce correct output. Same bytes.
Slide 8

Live Demo - demo.c

  • Run both commands live. Open a terminal and do it in front of the class - this is the most impactful moment of the lesson.
  • Before running, ask: "What do you expect to happen when we pass a .c file to Python?" Most will say "error". Let them be surprised.
  • Emphasize: Python doesn't care about the .c extension. It reads file content, not filenames. Same with GCC.
  • After the demo, ask: "Where would you even start if you wanted to write one of these yourself?" - set up the exercises slide.
Commands:
python examples/demo.c
gcc examples/demo.c -o demo && ./demo
Slide 9

Bash + Python

  • The subtlety here is quoting in bash. In shell, double-quoting a word doesn't prevent it from being executed as a command - it just prevents globbing and word splitting. So "exec" is still the exec builtin.
  • In Python, two adjacent string literals like "exec" "python3" are silently concatenated at compile time into one string - this is a little-known Python feature.
  • The $0 and $@ are NOT expanded by Python - they're literal dollar signs. Only bash expands shell variables.
  • When bash hits that line, exec replaces the entire shell process with Python - bash is gone and Python takes over running the same file from the top.
Commands:
bash examples/bash_python.sh
python3 examples/bash_python.sh
Slide 10

polycompiler - Overview

  • Real npm package - 304 GitHub stars as of this lesson. Made by a solo developer (evanzhoudev) as an experimental project.
  • Python and JS are much harder to polyglot than C+Python because they share so much syntax - you can't just hide one in the other's comment.
  • The trick needed a completely different approach: find an operator that means different things in both languages.
  • The output file must be .py.js - Node.js refuses to execute files with other extensions in certain module modes.
  • Point students to the GitHub repo - the README is very readable and explains the "aha moment" the author had.
Slide 11

polycompiler - The Live Trace

  • This is the most technically dense slide. Slow down here.
  • // is the pivot point. In Python it's an operator - both sides are evaluated. In JS it's a comment - everything after it on that line disappears.
  • JavaScript labeled statements (label: statement) are a rarely-used feature - most JS developers have never written one. lambda: is a perfectly valid label in JS because lambda is not a reserved word there.
  • In Python, lambda: starts a zero-argument lambda expression. Since it's never followed by a call (), the lambda object is created and immediately discarded.
  • Ask: "If you were designing a polyglot tool for Python and Ruby, what operator duality would you look for?"
Commands:
python3 examples/python_js.py.js
node examples/python_js.py.js
Slide 12

Why Does This Matter?

  • CTF: If students ever compete in capture-the-flag competitions, polyglot file forensics is a near-guarantee. Knowing to check for hidden ZIP/RAR/7z appended to images is a fundamental skill.
  • Security: PHAR (PHP Archive) polyglot attacks were used in real-world breaches as recently as 2022 against PHP apps with file upload features.
  • Parser literacy: Being able to read a language spec and identify "ignored regions" is the same skill used to find security bugs in parsers - this is how XSS and injection bugs get found.
  • Keep this slide quick - it's a survey. The exercises slide is where they engage.
Slide 13

Try It Yourself

  • Give them actual time to work - even 5-10 minutes here is valuable.
  • Exercise 1 is deliberately trivial - just running the demos. Gets everyone in the room to see it work with their own eyes.
  • Exercise 2 (extend demo.c) is the best learning exercise. Students have to understand both languages' ignored regions before they can add to the polyglot.
  • Exercise 3 (binary polyglot) is the most satisfying - it feels like a magic trick when the image still opens after unzipping.
  • The bonus (Python+HTML) is: use """...<!-- """ to hide the HTML header and <!-- ... --> to hide the Python footer. See examples/html_python.html.
Slide 14 - Build It Live

Complete Solution (keep off screen!)

  • Give students ~5 minutes to attempt it before revealing anything.
  • Common mistakes: forgetting the return 0;, putting Python code inside the C block, or not closing the triple-quoted string before print().
  • If someone finishes early, give them the stretch goal: add a greet() function in C and a matching def greet() in Python inside their respective sections.
Full working solution: 
#if 0
x = """
#endif

#include <stdio.h>

int main(void) {
    printf("Hello from C!\n");
    return 0;
}

#if 0
"""
print("Hello from Python!")
#endif
Stretch solution (with name function): 
#if 0
x = """
#endif

#include <stdio.h>

void greet(const char *name) {
    printf("Hello, %s! (from C)\n", name);
}

int main(void) {
    printf("Hello from C!\n");
    greet("HacKSU");
    return 0;
}

#if 0
"""
def greet(name):
    print(f"Hello, {name}! (from Python)")

print("Hello from Python!")
greet("HacKSU")
#endif
